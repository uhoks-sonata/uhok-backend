# Order Payment Research

## 목적

현재 `services/order` 기준 결제 흐름을 정리하고, 인메모리 개발환경에서 어떤 목표가 현실적인지 판단한다.

이 문서는 다음 질문에 답하기 위한 정리다.

- 현재 구현이 실제로 어떻게 동작하는가
- 제안하는 v2 웹훅 구조가 왜 더 실무적합한가
- 다만 현재 개발환경에서 어디까지를 "현실적인 목표"로 잡아야 하는가
- 이 코드베이스에서 무엇이 작은 수정이고, 무엇이 실제로는 범위가 큰 수정인가

## 현재 코드 기준 관찰

### 1. v1은 폴링 기반 레거시 흐름이다

`payment_v1_crud.py` 기준 v1은 아래 순서로 동작한다.

1. 주문 접근 검증
2. 주문 총액 계산
3. 외부 결제 생성 요청
4. `payment_id` 수신
5. 하위 주문 상태를 `PAYMENT_REQUESTED` 로 변경
6. 외부 결제 상태를 폴링
7. 완료면 `PAYMENT_COMPLETED`, 실패/타임아웃이면 주문 취소

즉, 설계 관점에서는 전형적인 polling 기반 결제 흐름이다.

하지만 현재 레포 안의 `payment-server` 구현은 `v2 webhook_auto_complete` 만 제공하고 있어, v1이 기대하는 `/pay`, `/payment-status/{payment_id}` 서버 구현은 레포 내부에서 확인되지 않는다.

따라서 v1은 "백엔드에 남아 있는 레거시 흐름"으로 보는 것이 맞다.

### 2. 현재 v2는 웹훅 기반이지만 완전히 비동기적이지는 않다

`payment_v2_crud.py` 와 `payment-server/routes/payment_routes.py` 기준 현재 v2는 아래처럼 동작한다.

1. Order API가 주문 검증과 금액 계산을 수행한다.
2. `tx_id`, `callback_url`, `callback_token` 을 생성한다.
3. 하위 주문 상태를 `PAYMENT_REQUESTED` 로 변경한다.
4. Order API가 Payment Server의 `POST /api/v2/payments` 를 호출한다.
5. Payment Server는 인메모리에 결제 상태를 `PENDING` 으로 저장한다.
6. Payment Server는 같은 요청 안에서 2초 대기한다.
7. Payment Server는 같은 요청 안에서 `PAYMENT_COMPLETED` 로 바꾼다.
8. Payment Server는 같은 요청 안에서 Order API 웹훅을 호출한다.
9. Webhook Handler가 주문 DB 상태를 반영한다.
10. `WaiterRegistry.resolve(tx_id, result)` 로 결과를 저장한다.
11. Order API는 이후 `subscribe()` 하면서 이미 저장된 결과를 즉시 받는다.
12. 클라이언트에 최종 응답을 돌려준다.

즉, 이름은 webhook 기반이지만 실제로는 "결제 생성 API 호출 하나가 내부적으로 완료와 웹훅까지 끌고 가는 구조"다.

### 3. WaiterRegistry는 실시간 대기열보다는 결과 캐시에 가깝다

현재 `WaiterRegistry` 는 다음 두 기능이 섞여 있다.

- callback token 검증용 임시 저장소
- 선도착 웹훅 결과를 저장해 나중 구독자에게 즉시 돌려주는 resolved cache

현재 호출 순서상 `subscribe()` 가 Payment Server 호출보다 먼저 걸리는 구조가 아니므로, 엄밀한 의미의 실시간 waiter 라기보다는 "웹훅 결과를 메모리에 저장해 두었다가 즉시 반환하는 장치"로 보는 것이 더 정확하다.

### 4. 최종 진실은 Payment Server가 아니라 Order DB다

Payment Server의 상태 저장소는 `payment_storage.py` 의 인메모리 딕셔너리다.

이 저장소는 다음 특징이 있다.

- 프로세스 메모리 기반
- 재시작 시 사라짐
- 단일 인스턴스 전제
- 개발/시뮬레이션 목적에는 충분

반면 실제 주문의 최종 상태는 Order API 웹훅 핸들러가 DB 상태 이력에 기록하는 값이다.

따라서 현재 구조에서도 진짜 source of truth 는 결제 서버 메모리가 아니라 Order DB 쪽이다.

### 5. 현재 v2에는 중요한 상태 충돌 가능성이 있다

현재 구현에서 가장 주의해야 할 부분은 timeout 처리와 늦은 webhook 처리의 관계다.

- Order API는 webhook 대기 timeout 시 주문을 즉시 `CANCELLED` 처리한다.
- 그런데 이후 `payment.completed` webhook 이 늦게 도착하면 현재 상태 검증 없이 다시 `PAYMENT_COMPLETED` 를 기록한다.

즉, 현재 코드에서는 `CANCELLED -> PAYMENT_COMPLETED` 같은 모순 이력이 생길 수 있다.

이 점은 구조 판단보다 먼저 정리해야 하는 현재 구현상의 핵심 리스크다.

## 결론

실무 적합성은 아래 순서로 판단한다.

1. 웹훅 중심 비동기 구조
2. 웹훅 중심 + 짧은 대기(waiter) 보조 구조
3. 현재 구현처럼 생성 요청 안에서 완료/웹훅까지 수행하는 구조
4. v1 polling 구조

즉, 방향 자체는 "웹훅 중심 구조"가 더 맞다.

다만 현재 환경이 인메모리라고 해서 곧바로 "즉시 응답 + 후속 조회" 구조를 아주 작게 끝낼 수 있는 것은 아니다. 현재 코드베이스에서는 그 변경이 API 계약과 프론트 흐름까지 건드리는 실제 범위가 있는 수정이다.

## 인메모리 환경 전제에서의 권장 구성

### 전제

다음 조건이면 인메모리 구성이 허용된다.

- 백엔드 인스턴스 1개
- Payment Server 인스턴스 1개
- 서버 재시작 시 상태 유실 허용
- 외부 분산 락/브로커 없음
- 로컬 개발 또는 통합 테스트 환경

이 조건에서는 `WaiterRegistry` 와 결제 저장소를 둘 다 인메모리로 둬도 된다.

단, 구조는 여전히 "웹훅이 최종 상태를 확정한다"는 원칙을 유지해야 한다.

### 추가로 반드시 필요한 현실 전제

현재 구조는 단순히 인메모리 여부만으로 성립하지 않는다. 아래 전제도 맞아야 한다.

- `WEBHOOK_BASE_URL` 이 payment-server 에서 실제로 접근 가능한 주소여야 한다.
- backend 와 payment-server 가 서로 도달 가능한 네트워크에 있어야 한다.
- 로컬 실행이든 Docker 실행이든 callback URL 경로가 실제 배포 형태와 맞아야 한다.

즉, 개발환경에서 웹훅 구조를 쓰려면 "메모리 저장소"보다 먼저 "콜백 URL 가시성"이 맞아야 한다.

### 권장 원칙

- Payment Server 메모리 상태는 임시 상태다.
- 최종 상태 확정은 Order API 웹훅 핸들러가 한다.
- `WaiterRegistry` 는 사용자 응답 편의를 위한 보조 장치다.
- 클라이언트 응답을 위해 짧게 기다릴 수는 있지만, 진실 저장소 역할을 맡기면 안 된다.

## 현재 구현에서 아쉬운 점

### 1. 결제 생성 API가 너무 많은 책임을 갖는다

현재 `POST /api/v2/payments` 는 아래를 한 요청 안에서 전부 수행한다.

- 결제 생성
- 2초 대기
- 완료 처리
- 웹훅 전송
- 실패 시 취소 처리

이 구조는 데모/로컬 테스트에는 편하지만, 실무에서는 생성 API와 최종 결과 확정을 분리하는 편이 낫다.

### 2. WaiterRegistry가 운영형 구조로 확장되기 어렵다

현재 `WaiterRegistry` 는 프로세스 메모리 기반이라 아래 상황에서 바로 한계가 생긴다.

- 백엔드 인스턴스가 여러 개일 때
- 서버가 재시작될 때
- 웹훅이 다른 인스턴스로 들어올 때
- 롱폴링 요청과 웹훅 요청이 다른 프로세스에 걸릴 때

로컬 개발에서는 괜찮지만, 운영 구조로 그대로 가져가면 안 된다.

### 3. 상태 전이와 커밋 경계가 분산되어 있다

상위 헬퍼는 일괄 상태 변경처럼 보이지만, 실제 하위 상태 변경 함수와 알림 생성 로직에서 각각 `commit()` 이 일어난다.

즉, 상위 흐름 입장에서는 "하나의 결제 상태 전이"가 여러 번의 커밋으로 분산된다.

개발 단계에서는 동작시킬 수 있지만, 운영 안정성을 강화할 때는 트랜잭션 경계를 다시 설계할 필요가 있다.

### 4. timeout 이후 fallback 경로가 아직 완성돼 있지 않다

문서상으로는 "짧게 기다리다 안 되면 조회로 전환"이 맞는 방향이다.

하지만 현재 코드베이스에는 다음 한계가 있다.

- `tx_id` 기준 후속 poll API가 실제로 구현되어 있지 않다.
- 프론트도 `PENDING` 분기는 일부 갖고 있지만, 실제 백엔드 롱폴링 API 부재를 전제로 임시 처리하고 있다.
- 현재 Order API v2 응답 스키마와 프론트 명세는 최종 완료형 응답을 기본 전제로 갖고 있다.

따라서 "즉시 `PENDING` 응답"은 좋은 방향이지만, 단순한 내부 리팩터링이 아니라 API 계약 변경에 가깝다.

## 개발환경에서의 현실적인 목표

현재 인메모리 개발환경에서 가장 현실적인 목표는 아래처럼 잡는 것이 맞다.

### 1단계 목표: 현재 구조를 모순 없이 정리

이 단계는 반드시 먼저 해야 하는 목표다.

- timeout 시 즉시 취소할지, 아니면 미확정 상태로 둘지 정책을 하나로 통일
- 늦게 도착한 webhook 이 기존 취소 상태를 덮어쓰지 않도록 방어
- `confirm/v2` 라우터가 내부 `HTTPException` 을 500으로 덮어쓰지 않도록 정리
- callback URL / 환경 변수 / 개발 실행 방식 문서화

즉, 지금 개발환경에서 제일 현실적인 목표는 "아키텍처 업그레이드" 이전에 "현재 v2 흐름의 모순 제거"다.

### 2단계 목표: 웹훅 중심 구조로 천천히 이동

그 다음 단계에서 아래 목표를 잡을 수 있다.

- `payment_storage.py` 인메모리 저장소 유지
- `WaiterRegistry` 인메모리 저장 유지
- 웹훅 서명 검증 유지
- callback token 검증 유지
- 웹훅을 통한 최종 상태 반영 유지

그리고 아래를 점진적으로 바꾼다.

- `POST /api/v2/payments` 는 즉시 `PENDING` 또는 `accepted` 수준의 응답만 반환
- 완료 처리는 요청 종료 후 백그라운드 작업으로 이동
- Order API는 필요 시 짧게만 기다림
- timeout 이후에는 상태 조회나 후속 확인 흐름으로 이어지게 설계

단, 이 단계는 backend 단독 수정으로 끝나지 않는다.

- 백엔드 응답 스키마 조정
- 라우터 계약 조정
- 프론트 후속 상태 확인 흐름 조정
- 필요하면 `tx_id` 또는 주문 단위 조회 경로 정리

즉, 방향은 맞지만 "최소 수정"이라고 부르기에는 범위가 생각보다 크다.

## 권장 흐름

### 장기적으로 맞는 권장 흐름

1. Client가 Order API v2로 결제 요청
2. Order API가 주문 검증 및 금액 계산
3. Order API가 하위 주문 상태를 `PAYMENT_REQUESTED` 로 변경
4. Order API가 Payment Server에 결제 생성 요청
5. Payment Server는 인메모리에 `PENDING` 저장 후 즉시 응답
6. Payment Server는 백그라운드 작업으로 완료 시뮬레이션 또는 외부 결제 완료 처리
7. 완료되면 Payment Server가 Order API 웹훅 호출
8. Webhook Handler가 Order DB를 `PAYMENT_COMPLETED` 또는 `CANCELLED` 로 반영
9. `WaiterRegistry` 에 결과가 있으면 현재 요청을 깨워 즉시 반환
10. 대기하지 않는 클라이언트는 상태 조회 API로 최종 상태 확인

### 이 흐름이 더 나은 이유

- 결제 생성 API가 오래 붙잡고 있지 않는다.
- Payment Server 응답 지연이 곧바로 사용자 지연으로 이어지지 않는다.
- 생성과 완료를 분리해 책임이 명확해진다.
- 웹훅을 최종 이벤트로 삼아 구조가 단순해진다.
- 나중에 Redis/DB 기반으로 옮기기 쉽다.

## 지금 코드베이스에서의 현실적인 해석

현재 코드베이스 기준으로는 아래처럼 해석하는 것이 가장 현실적이다.

### 유지해도 되는 것

- Payment Server 상태 저장소를 인메모리로 두는 것
- `WaiterRegistry` 를 인메모리로 두는 것
- 웹훅 기반 최종 상태 반영 원칙

### 바로 고쳐야 하는 것

- timeout 시 즉시 취소하는 현재 정책과 늦은 webhook 완료 허용 가능성의 충돌
- 상태 전이 모순 가능성
- 라우터의 예외 전달 방식
- callback URL 전제 누락

### 범위가 더 큰 변경으로 봐야 하는 것

- `POST /api/v2/payments` 즉시 `PENDING` 응답화
- Order API `confirm/v2` 를 최종 완료형 응답에서 accepted/async 흐름으로 전환
- 프론트의 완료 대기 및 후속 조회 UX 정리
- `tx_id` 기반 후속 조회 API 신설 또는 주문 상태 조회 흐름 재정의

## 운영 전환 시 확장 방향

운영으로 갈 때는 아래 순서로 교체하면 된다.

1. `WaiterRegistry` -> Redis
2. Payment Server 상태 저장소 -> Redis 또는 실제 DB
3. 백엔드 다중 인스턴스 허용
4. 웹훅 멱등성 강화
5. 상태 전이 트랜잭션 경계 정리

즉, 인메모리 개발환경에서도 처음부터 구조 방향은 운영형에 맞춰두되, 저장소만 단순화하는 것은 맞다. 다만 현재 레포에서는 timeout 정책과 API 계약을 먼저 정리하지 않으면 그 방향 전환이 안전하게 이어지지 않는다.

## 최종 정리

현재 환경이 인메모리라면 큰 방향은 아래가 맞다.

- 결제 서버 상태 저장소: 인메모리 허용
- WaiterRegistry: 인메모리 허용
- 최종 상태 저장소: Order DB
- 구조 방향: 웹훅 중심 유지

하지만 개발환경에서 "지금 당장 현실적인 목표"는 아래 순서가 맞다.
